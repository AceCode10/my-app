The Systems Life Cycle

Topic 7 · Sections 7.1 – 7.6

7.1 The Systems Life Cycle

A structured process for developing, implementing and maintaining computer systems.

1. Analysis

Study current system; identify problems; gather requirements

2. Design

Design file structures, input/output formats, validation

3. Development & Testing

Build and test the system with test data

4. Implementation

Switch from old to new system using a chosen method

5. Documentation

Create technical and user documentation for the system

6. Evaluation

Assess efficiency, ease of use and suitability against original requirements

7.1 Analysis — Research Methods

The current system is studied using four techniques:

MethodDescriptionAdvantagesDisadvantages
Observation Watch users interact with the system to understand its workings First-hand, unbiased information Time-consuming; may not reveal all issues
Interviews Direct one-to-one conversations with users about their experience In-depth exploration of issues Expensive; time-consuming; no anonymity may affect responses
Questionnaires Predetermined questions given to users to gather opinions Quantitative analysis; efficient; answers given quickly Limited by predetermined questions; low response rates; anonymity may lead to exaggerated answers
Examination of Existing Documents Review system documentation, user guides, reports Insights into system history; reveals unknown issues May be outdated; incomplete; time-consuming; expensive

Analysis — Requirements & Specification

Recording & Analysing the Current System

  • Identify data input, processed and output by the current system
  • Identify problems with the current system — what could be improved?
  • Identify user requirements — what does the new system need to do?
  • Identify information requirements — data the new system must process
The Systems Analyst creates a Requirements Specification outlining improvements and expectations for the new system.

System Specification — Hardware & Software

Hardware to consider:

  • Barcode readers, scanners, touch screens
  • 3D printers, monitors, speakers

Software to consider:

  • Operating system
  • Applications software
  • Size and type of storage needed

Choices justified based on: system requirements, compatibility, costs, user needs, system performance.

7.2 Design

File/Data Structures

  • Field length — number of characters allowed in a field
  • Field name — identifier for the field
  • Data type — specifies kind of data (text, number, date)
  • Coding of data — e.g. M for male, F for female

Input & Output Formats

  • Data capture forms: paper-based or electronic; collect data in structured format
  • Screen layouts: how information is presented on screen
  • Report layouts: how information is organised in printed/digital reports

Validation Routines

Examines data submitted to determine if it meets specific requirements — automatic check performed by programming.

  • Range check — data within a specified range of values
  • Character check — data contains only allowed characters
  • Length check — data is of a specified length
  • Type check — data is the correct data type
  • Format check — data conforms to a specific format
  • Presence check — data is present and not blank
  • Check digit — digit added to verify accuracy of a number

7.3 Development & Testing

Test Strategies

  • Test each module — verify individual components function correctly
  • Test each function — ensure all features work correctly
  • Test the whole system — confirm overall performance and integration

Test Plan Components

  • Test data — specific data used for testing
  • Expected outcomes — predicted results from test data
  • Actual outcomes — results obtained from testing
  • Remedial action — steps taken to fix identified issues

Types of Test Data

Example used: months in a year (valid: 1–12)

  • Normal data — valid, expected values within acceptable range
    e.g. any whole number 1–12
  • Abnormal data — invalid or unexpected values:
    • outside range: 0, -6, 13, 15
    • wrong type: "July", 3.5, 4.2
  • Extreme data — values at the limits of acceptability
    e.g. 1 or 12

Live data: data already used with current system — results already known, used to verify new system gives same results.

7.4 Implementation — 4 Methods

MethodDescriptionUsed WhenAdvantagesDisadvantages
Direct Changeover Old system replaced immediately by new system Quick implementation necessary Fast; cost-effective (one system only) High risk of failure; no fallback; users can't train beforehand
Parallel Running Both old and new systems run simultaneously until old is phased out Smooth transition with minimal risk needed Lower risk; easy to compare systems Time-consuming; resource-intensive (double the work)
Pilot Running New system implemented in one small department/branch first Testing in real-world setting before full rollout Low risk; allows fine-tuning; staff training time; few errors Slower; potential inconsistencies; two systems in use cause confusion; no backup for pilot department
Phased Implementation New system implemented in stages, each replacing part of old system Gradual transition preferred to minimise disruption Reduced risk; easier to manage Takes longer; potential compatibility issues between stages

7.5 Documentation — Technical

Technical documentation — detailed information on the system's inner workings for developers and IT staff. Used to maintain, repair and update the system.

Components Include:

  • Purpose of the system/program
  • Limitations of the system
  • Program listing (source code)
  • Program language used
  • Program flowcharts/algorithms
  • System flowcharts
  • Hardware and software requirements
  • File structures
  • List of variables (names and purposes)
  • Input format
  • Output format
  • Sample runs/test runs
  • Validation routines

7.5 Documentation — User

User documentation — instructions and guidance for end-users on how to operate the system.

Components Include:

  • Purpose of the system
  • Limitations of the system
  • Hardware and software requirements
  • How to load/run/install software
  • How to save a file
  • How to print data
  • How to add records
  • How to delete/edit records
  • Input format
  • Output format
  • Sample runs
  • Error messages and their meanings
  • Error handling steps
  • Troubleshooting guide/helpline
  • Frequently Asked Questions (FAQ)
  • Glossary of terms

7.6 Evaluation

Measures a system's productivity, efficiency and compliance with its goals to identify strengths, weaknesses and development areas.

Efficiency

Analyse time, money and resource use. Is the system performing at its best?

Questions: Does it operate faster than old system? Does it reduce staff time? Does it reduce costs?

Ease of Use

Check usability and accessibility. Is it simple to understand? Can users complete tasks easily?

Questions: Can users make bookings easily? Can staff use it with minimal training?

Suitability

Compare solution against original task requirements. Does it meet user and business needs?

Questions: Is it suitable for all departments? Does it meet customer needs? Does it match original requirements?

Also: Collect and examine user feedback from testing. Identify limitations and suggest necessary improvements.

Topic 7 — Key Points

  • Analysis: Observation · Interview · Questionnaire · Document examination → Requirements Specification
  • Design: Field structures · Data types · Input/output formats · 7 validation routines
  • Testing: Module · Function · System tests · Normal / Abnormal / Extreme data · Live data
  • Implementation: Direct (fast, risky) · Parallel (safe, costly) · Pilot (low risk, slow) · Phased (gradual)
  • Technical Docs: For developers — code, flowcharts, file structures, test runs
  • User Docs: For end-users — how to use, error messages, FAQ, glossary
  • Evaluation: Efficiency · Ease of Use · Suitability vs original requirements · User feedback · Identify limitations
